§0 One-line summary

A local collaborative attack-chain workbench where researchers and AI agents both author MITRE-aligned kill chains and attach rich evidence (Frida traces, HAR, PCAP, source artifacts, sandbox screenshots). Three intake paths feed the corpus: (1) markdown reports from prior investigations are ingested into chains by an agent, (2) new apps arrive via a Pixel 7 broker device as job tickets and are auto-analyzed by an agentic worker, (3) knowledge-base indicators (benign + malicious pools of packages / domains / ELF hashes) bias every investigation. Every researcher decision (agree · disagree · edit · flip · verify) — at the chain verdict, node, or evidence level — feeds a labeled corpus that agents read back to improve over time.

§5 Agent learning loop

The pipeline that turns researcher judgment into reusable signal. Every accept/reject/edit/verify writes a label row that's exportable as JSONL — that's the training data for the next-generation agents.

Why labels (not raw audit) drive learning

The audit_log records what changed. The labels table records what a researcher decided about an agent's claim — that's the supervision signal. Keeping them separate means: (a) we can purge audit noise without losing training data, (b) labels naturally include before/after diffs and the source agent, (c) the JSONL export is already in the shape an eval harness wants.

Where retrieval fits

Today retrieval is pure SQL TF-IDF over node_terms. When agents need "show me prior chains like this candidate" before they propose, they call GET /api/similar/:nodeId. Phase 4+ swaps that for sqlite-vec + MiniLM embeddings without touching the API contract.

§6 Report ingestion subsystem

A pool of .md reports — one report per investigated artifact or campaign — is the primary historical-knowledge intake. Each report carries a declared category and (optionally) a declared verdict tp | fp. The reviewer can later confirm or flip the verdict; flipped decisions are the highest-signal training data we have.

-- N apps per artifact (one per version)

  id              TEXT PRIMARY KEY,
  app_id          TEXT REFERENCES apps(id),   -- NULLABLE for legacy
  source_path     TEXT,             -- path on inbox
  filename        TEXT,
  body_blob_sha   TEXT,             -- ref to blobs store (immutable)
  declared_category TEXT,           -- riskware/toll_fraud/phishing
  declared_label  TEXT,             -- 'tp' | 'fp' | NULL
  effective_label TEXT,             -- after reviewer (NULL until reviewed)

  status          TEXT,             -- pending|processing|ingested|rejected
  ingested_chain_id TEXT,           -- fk → chains.id
  imported_at     TEXT,
  processed_at    TEXT,
  reviewed_at     TEXT,
  reviewed_by_id  TEXT,
  content_hash    TEXT              -- for idempotent re-ingest on .md edit
);
CREATE INDEX idx_reports_app ON reports(app_id);
CREATE INDEX idx_reports_artifact ON reports(app_id);

"flipped" semantics — why it matters

Most labels say "agent guessed X, reviewer agreed". A flipped label says "the original source called this TP, but on inspection it's FP" (or vice versa). That's worth more than 10 regular labels because it teaches:

  • which surface features are misleading
  • which classifiers / heuristics over-trigger
  • which reports / sources skew which direction

We surface flipped reports as a dedicated "flip review" lane in the dashboard so disagreements with prior labelings are visible, not buried in the audit log.

§7 Device broker — Pixel 7 as portable job carrier

A Pixel 7 is the physical interface between two machines: the field machine (a separate system where apps are installed/observed and where job tickets are produced) and the darkseed machine (where analysis happens). The device is physically carried between them. Connection over USB-C; transfers via adb pull / adb push. Single-writer per directory: field system owns jobs/inbox/; darkseed owns results/outbox/ and kb/. One machine ↔ one device at a time — no concurrent sharing. Paths are placeholders for now.

§8 Indicator knowledge base (benign + malicious pools)

Two sided collections of known indicators that bias every agent investigation. Hits don't auto-verdict — they raise confidence and surface as priors in the chain. A package matching the benign pool isn't ignored; it's flagged "known good, double-check before flagging." A domain matching the malicious pool isn't auto-malicious; it's "known bad, this raises the floor."

§11 Key design trade-offs

SQLite, not Postgres

Single-file, zero-ops, FTS5 + vec extensions, fast for 100k chains. Trade-off: no concurrent writers beyond ~one process; we'll have to migrate if we ever need multi-instance writes (or front it with a write-coordinator).

Content-addressed blobs on disk, not in SQLite

Deduplication is free (same APK uploaded twice = one file). SQLite stays small. Trade-off: backup story is "copy data/ entire" — not "dump one db file."

Discriminated-union evidence schema

Each evidence kind has its own zod payload + own React viewer. Adding a new kind is two files; no migration. Trade-off: 20+ tiny viewer functions to maintain — but each is <50 lines and rarely changes.

TF-IDF before neural embeddings

Zero deps, pure SQL, instant warm-up. Good baseline retrieval for surface vocabulary (IOC + technique). Trade-off: misses semantic-only matches (e.g., "SMS hijack" ↔ "intercept OTP"). Vector store closes that gap once we add it.

Proposals are first-class (status enum), not a separate queue table

A proposed node lives in chain_nodes with status='proposed'. So the graph can render proposals inline (with the ⚑ badge) and the reviewer can see context. Trade-off: every query needs a status filter — easy to forget. Mitigated by always-default-filtering in the repo layer.

Agents and humans use the same API

Only difference: x-actor-kind header. No special agent endpoints, no special data shape. Trade-off: no "agent superpowers" — both are equally constrained by validation/auth. That's a feature.
