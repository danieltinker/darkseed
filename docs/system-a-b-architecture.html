§0 What this doc is

The joint architecture that binds the existing two-agent malware-analysis system (described in jetski-sync) with darkseed (the persistent brain DB at this repo). It supersedes nothing inside either system internally — it specifies the contracts where they meet.

Two agents, two host machines, one physical device shared by both, one persistent brain on the Agent B side. Reports stay incomplete through scoring → static → dynamic, then the operator marks them complete after reviewing assembled evidence.

System AmacOS — Agent A (static) + daily-driver dashboard doesqueue intake, scoring, install, slice (JADX + Ghidra + native), rubric hunt, playbook + hooks + decryptors, joint-report assembly from B's output surfacesreports list (state + verdict pills), minimal proof viewer per app
System BWindows 11 (Claude orchestrator) → SSH → Kali VM doesdynamic execution via Frida + multi-VPN baseline + HTTP Toolkit (opt) + hook synthesis when A is silent surfacesvia darkseed (Flask ops view absorbed into darkseed)
darkseedon Kali, same host as Agent B rolepersistent brain — chains, evidence, KB, similarity, audit, labels export. Receives joint markdown reports from the publisher. phasearchive-only for PoC; agent priors deferred to v2

§3 Report lifecycle (strict state machine)

Every app under investigation lives in exactly one state at any time. Each transition is gated by an evidence check. The operator's Reports tab renders state as a colored pill.

owned by Agent A owned by Agent B device transit darkseed processing operator decision

Transition gates (each requires evidence)

scoring → score_completeDARBEK scrape returns a score row + metadata
score_complete → awaiting_slicescore ≥ threshold (configurable; below → archived)
awaiting_slice → slicingoperator clicks Slice; JADX + Ghidra + native extractors invoked
slicing → sliceddecompile output exists; tarball is sha-computed
sliced → hunt_runningoperator clicks Run Agent A Hunt
hunt_running → hunt_completeriskware_comprehensive_report.md + execution_playbook.json exist locally
hunt_complete → awaiting_dynamiceverything pushed to device; state.A_complete written
awaiting_dynamic → dynamic_runningstate.B_started appears (cable now on Kali)
dynamic_running → dynamic_completestate.B_complete + dynamic_results.json exist on device
dynamic_complete → awaiting_pickupMac ADB regains device; results pulled
awaiting_pickup → publishedpublisher renders joint .md into darkseed inbox + blobs stored
published → in_reviewingester succeeded; chain exists in darkseed; A's Proofs tab populated
in_review → verifiedoperator sets verdict (TP or FP) in A's dashboard; mirrored to darkseed
* → erroredstate.A_error or state.B_error at any point

Cable-move tolerance

The cable is moved by a human between Mac and Kali. The state machine handles brief device-offline windows:

  • Mac's collector polls every 5s for state files; when device disconnects, the row pauses at its current state, no transition fires.
  • Kali's dyn-playbook-scout polls the same dir every 5s once cable arrives; sees state.A_complete, claims state.B_started.
  • The awaiting_dynamic and awaiting_pickup states have a configurable TTL (default 30min, soft warn). Operator can override.
  • If a job is left in awaiting_* > TTL, it surfaces on both dashboards as stuck with the actor's last-seen heartbeat.

§7 darkseed schema additions (Phase 10 migration)

SQL changes needed to model the joint system. Forward-compatible with the existing schema; nothing in Phases 1-8 changes.

-- migrations/005_a_b_integration.sql

-- 1. Extend reports lifecycle states (DECIDE STATES)
-- Existing report.status: pending | processing | ingested | rejected
-- Add: scoring | score_complete | awaiting_slice | slicing | sliced
--      | hunt_running | hunt_complete | awaiting_dynamic
--      | dynamic_running | dynamic_complete | awaiting_pickup
--      | in_review | verified | flipped | inconclusive | errored | archived

ALTER TABLE reports ADD COLUMN static_score INTEGER;
ALTER TABLE reports ADD COLUMN verified_score INTEGER;
ALTER TABLE reports ADD COLUMN score_delta INTEGER;
ALTER TABLE reports ADD COLUMN static_self_sufficient_a TEXT;   -- yes | no
ALTER TABLE reports ADD COLUMN static_self_sufficient_b TEXT;   -- yes | no

-- 3. App augmentations
ALTER TABLE apps ADD COLUMN marmot_id TEXT;
ALTER TABLE apps ADD COLUMN install_source TEXT;                -- DECIDE-10
ALTER TABLE apps ADD COLUMN manifest_summary_blob_sha TEXT;     -- EXT-1

-- 4. Chains carry the evasion delta + verification
ALTER TABLE chains ADD COLUMN evasion_delta_json TEXT;          -- DECIDE-A4
ALTER TABLE chains ADD COLUMN vpn_exits_tested TEXT;            -- JSON array
ALTER TABLE chains ADD COLUMN agent_a_static_score INTEGER;
ALTER TABLE chains ADD COLUMN agent_b_verified_score INTEGER;

-- 5. IOCs carry per-observation VPN + phase
ALTER TABLE iocs ADD COLUMN vpn_state TEXT;                     -- DECIDE-1
ALTER TABLE iocs ADD COLUMN rubric_id TEXT;
ALTER TABLE iocs ADD COLUMN phase INTEGER;                      -- DECIDE-4

-- 6. Nodes know which rubric phase they came from
ALTER TABLE chain_nodes ADD COLUMN rubric_id TEXT;
ALTER TABLE chain_nodes ADD COLUMN phase INTEGER;
CREATE INDEX IF NOT EXISTS idx_nodes_rubric ON chain_nodes(rubric_id);

-- 7. Workflows (DECIDE-14)
CREATE TABLE IF NOT EXISTS workflows (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  rubric_id TEXT NOT NULL,
  package_sha TEXT,                  -- nullable when generic across packages
  name TEXT NOT NULL,
  description_md TEXT,
  trigger_steps_json TEXT NOT NULL,   -- the ordered (hook + adb + mock) tuple
  success_count INTEGER NOT NULL DEFAULT 1,
  last_used_at TEXT NOT NULL DEFAULT (datetime('now')),
  source TEXT,                       -- 'auto:from_chain:<chain_id>' | 'manual'
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  created_by_kind TEXT, created_by_id TEXT
);
CREATE INDEX IF NOT EXISTS idx_workflows_rubric ON workflows(rubric_id);

-- 8. Reviewer-facing per-app summary view (denormalized for the dashboard)
CREATE VIEW IF NOT EXISTS reports_view AS
SELECT r.id, r.app_id, r.status,
       r.static_score, r.verified_score, r.score_delta,
       r.declared_label, r.effective_label, r.flipped,
       a.package_name, a.marmot_id, a.install_source,
       c.id AS chain_id, c.verdict, c.severity,
       r.imported_at, r.reviewed_at
FROM reports r
LEFT JOIN apps a ON a.id = r.app_id
LEFT JOIN chains c ON c.source_report_id = r.id;

§10 Hook synthesis + write-back loop

DECIDE-A1 — full autonomy + full transparency. Agent B's dyn-hook-synth may produce a Frida hook whenever Agent A's hook is silent or doesn't cover the phase being proved. New IOCs round-trip back into A's rubric on the next investigation cycle.

§11 Score recompute — static + verified

DECIDE-A4. Two numbers, both visible. Verdict is independent.

static_score (Agent A, immutable)

Sum of (rubric_points × confidence_weight) over A's rubrics, capped at 120. Confidence: High=1.0, Medium=1.0, Low=0.0 (Low items only log as inconclusive signals — 0 pts).

Written once in execution_playbook.json:job_metadata.static_score, copied into reports.static_score. Never modified.

verified_score (Agent B, mutable)

Per-rubric verification fires verification_result evidence: confirmed keeps the rubric's points, refuted zeroes them, inconclusive halves them. NEW rubrics that B discovered (e.g., via synth hook) ADD their points.

Lives in dynamic_results.json:verified_score + delta. Persists to reports.verified_score + chains.severity_score.

UI shows: 24 → 28 (+4). Operator sees what A guessed vs what B proved.

§12 Verdict mirror (A → darkseed)

Operator sets verdict in Agent A's dashboard. A POSTs to darkseed so the chain's verdict + labels.jsonl stay synced. PoC: HTTP API on local network with auth-token bearer header.

// POST kali:3001/api/chains/<chain_id>/verdict
// (Agent A's dashboard on Mac → darkseed on Kali)
Headers:
  x-actor-kind: user
  x-actor-id: <operator_email>
  x-mirror-from: agent_a_dashboard
  authorization: Bearer <darkseed_pat>        # configured per-operator
Body:
{
  "verdict": "malicious",                       # malicious | benign | inconclusive | pending
  "notes": "operator reasoning markdown",
  "static_score": 24,
  "verified_score": 28,
  "set_by_dashboard": "agent_a"
}
// darkseed responds:
{ "ok": true, "flipped": true, "label_id": 42 }

Cable-mode fallback

If darkseed is unreachable when the operator sets the verdict, A's dashboard queues the verdict locally. Next time the cable lands on Kali and the operator (or a cron) syncs, A POSTs queued verdicts to darkseed in order. The chain's verdict_set_at uses the original click time, not the sync time.

§13 Agent A's minimal proof viewer (the only new UI on A's side)

Six tabs. Daily-driver for the operator. No deep brain features — those live in darkseed.

Tab 1 — Lock Queue

Live list of locked apps from DARBEK (or whatever pluggable intake source). Manual: Refresh. Auto on a 5-min cron. Each row: package, marmot_id, lock state, score (when computed).

Tab 2 — Reports

Two pills per row: state (scoring → … → verified) and verdict (TP / FP / pending / flipped). Sortable by score, age, state. Filter by state or verdict.

Tab 3 — Install + Slice

Per locked app, Slice button. Synchronous: spawns JADX + Ghidra + native + framework detectors. Shows live log. On success: tarball sha computed, app row state → sliced.

Tab 4 — Hunt

Run Agent A Hunt button per sliced app. Triggers Agent A's rubric analysis. Shows running status, rubric matches as they fire, evidence file count. On success: state → hunt_complete. Operator told to move cable to Kali.

Tab 5 — Proofs (minimal)

Per app in in_review: rendered joint_report.md (the same one darkseed got), with inline thumbnails for screenshots, expandable Frida ndjson timeline, HAR waterfall summary, per-VPN IOC chips. Read-only — no editing here.

Tab 6 — Verdict

Per app, 4 buttons (TP / FP / inconclusive / pending) + notes textarea. On click: POSTs to local FS report.md AND darkseed (with cable-mode fallback per §12). State → verified or flipped.
